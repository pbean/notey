## 1. How Developers Take Notes and Capture Information During Coding Sessions

### What Developers Capture

Developers maintain several distinct types of notes during coding work:

- **Running notes / coding journals**: Step-by-step documentation of work in progress, including code snippets, shell commands, links to documentation, mistakes, explorations, and detours. These are "dynamic coding journals" capturing what was tried, the output, and insights gathered.
- **Topical notes**: Self-contained notes on individual topics -- either research/learnings (e.g., "how to structure REST API endpoints") or recipe-style snippets (e.g., "how to cherry-pick commits in git").
- **Scratchpad / throwaway content**: Quick experiments, Slack messages drafted before sending, JSON API responses, meeting notes, daily to-do lists. Most developers maintain some form of "quick way of doing experiments" with titles like "dummy", "scratchpad", or "junk."

### How Developers Access Knowledge

From the Stack Overflow 2025 Developer Survey (49,000+ responses, 177 countries):
- Stack Overflow (84%), GitHub (67%), and YouTube (61%) are the leading knowledge resources
- 68% of new developers use technical documentation more than other resources
- When visiting Stack Overflow, developers' top activity is "reading comments" -- they want human context, not just answers

### Productivity Context (JetBrains Developer Ecosystem 2025, 24,534 developers)

- 89% say non-technical factors (job design, clear communication, peer support) influence their productivity
- 84% say technical factors (tool performance and reliability) influence their productivity
- 66% of developers don't believe current metrics reflect their real contributions
- Developers consistently request greater transparency and clarity in how their work is measured

## 2. Developer Tool Adoption Patterns -- Tool Fatigue & Consolidation

### Scale of Tool Sprawl

From a 2025 survey of 300 IT professionals (U.S. and Western Europe):
- **75% lose 6-15 hours weekly** to tool sprawl -- navigating an average of **7.4 tools** to build applications
- **44% of teams** juggle 10 or more DevOps tools, with many overlapping in functionality
- Developers juggle an average of **14 tools** total, prompting 62% of executives to prioritize consolidation
- **$1 million annually** per development team lost to tool sprawl productivity costs

From a separate survey of 135+ engineers:
- **52% flag context switching** as a major productivity drain
- **56% report tool overload** affects their performance every week
- Workers toggle between apps **33 times per day** on average

### AI Tool Fatigue (2025-2026)

- Product Hunt lists **30+ new AI tools every day**; 5-10 are developer-focused
- 85% of developers regularly use AI tools (JetBrains 2025), 84% use or plan to use them (Stack Overflow 2025)
- But **only 3.1% "highly trust"** AI tools, and 46% don't trust AI output at all
- Positive sentiment for AI tools **decreased from 70%+ (2023-2024) to 60% (2025)**
- 66% of developers struggle with AI solutions that are "close but ultimately miss the mark"
- 45% say debugging AI-generated code takes longer than writing it themselves

### Consolidation Trends

- **74% reported MORE tool sprawl after migration**, not less (CloudBees 2025 DevOps Migration Index)
- 61% of DevOps leaders report migration fatigue prevented new initiatives for 6+ months
- **94% of developers are dissatisfied** with current toolsets
- Platform engineering projected to reach **80% adoption by 2026** (Gartner) as a coping strategy
- Learning resource usage is consolidating: developers using 8+ learning resources dropped from 49% (2024) to 7% (2025)

## 3. "Developer Scratchpad" / "Developer Notepad" Search Trends & Community Discussions

### Hacker News

- **Heynote's Show HN** (Dec 2023) received **978 points and 294 comments** -- overwhelmingly positive reception. It ranked in Best of Show HN lists.
- **Stashpad's Show HN** (2022) also generated strong interest as "a new scratchpad for developers."
- Developer scratchpads remain a perennial topic; the concept of a dedicated quick-capture tool resonates strongly.

### General Community Sentiment

- Developers commonly maintain ad-hoc scratchpad projects ("dummy", "throwaway", "junk") -- these replace Google searches and are "almost as quick as a search with less potential for wrong or misleading information."
- The category is fragmented: developers use everything from untitled Notepad++ tabs to VS Code scratch files to full-blown Obsidian vaults. There is no dominant purpose-built solution.
- Windows Notepad itself is modernizing (2025-2026): tabs, autosave, Markdown support, and AI features (Rewrite/Summarize) -- validating that even Microsoft sees demand for the lightweight capture use case.

### GitHub Topic: "scratchpad"

The GitHub "scratchpad" topic continues to show active development with multiple new entries. The space is clearly active but no single tool has achieved category dominance beyond Heynote.

## 5. New Developer Note-Taking / Scratchpad Tools (2025-2026)

### Terminal / CLI-First Tools

- **Thoth** -- Terminal scratchpad directly inspired by Heynote. Persistent text buffer saved as markdown in home directory. Integrates with Neovim, Helix, Vim, and other terminal editors. Targets developers who never leave the terminal.
  - Source: [Thoth GitHub](https://github.com/jooaf/thoth)

- **Snip** -- Fast CLI note-taking tool for developers. Create, search, and manage notes from the terminal. Built with Go for speed.
  - Source: [Snip](https://snip-notes.vercel.app/)

- **z-scratchpad** -- Lightweight Go-based tool for notes/wiki. Uses plain text files with minimal syntax on a plain filesystem. Documents searchable/editable with nano, grep, sed. Pre-release state.
  - Source: [z-scratchpad GitHub](https://github.com/volution/z-scratchpad)

### Desktop GUI Tools

- **LinNote** -- Fast, keyboard-driven scratchpad for Linux. Built for KDE Plasma with Qt6, featuring markdown support and smart calculations.
  - Source: [LinNote - LinuxLinks](https://www.linuxlinks.com/linnote-scratchpad/)

- **Antinote** -- Minimalist note-taking app acting as a quick-access scratchpad that floats on screen. Mac-focused.
  - Source: [12 New Mac Apps Worth Trying in 2026](https://www.gulfnews.ca/12-new-mac-apps-worth-trying-in-2026-to-boost-productivity-and-creativity/)

- **ScratchPad 2.0** (February 2026) -- Complete rewrite from the ground up using Swift and SwiftUI. Universal app supporting Intel and Apple Silicon Macs.
  - Source: [ScratchPad 2.0 Blog Post](https://blog.alexseifert.com/2026/02/20/scratchpad-2-0-over-a-decade-later/)

### Established Players Still Active

- **Stashpad** -- Hierarchical notepad with keyboard-shortcut emphasis. "Helps developers stay in flow and easily capture notes, ideas, or thoughts as they switch between tasks."
  - Source: [Stashpad on Product Hunt](https://www.producthunt.com/products/stashpad-2)

- **Codex** -- Note-taking for programmers with rich code support.
  - Source: [Codex Notes](https://codexnotes.com/)

- **Supernotes** -- Fast card-based note-taking with Markdown, LaTeX, collaboration features.
  - Source: [Supernotes for Developers](https://supernotes.app/notetaking-for-developers/)

### Key Market Gap Observation

No tool in this space currently combines ALL of: (1) global hotkey instant capture, (2) terminal CLI integration, (3) workspace/project awareness, (4) lightweight native performance (non-Electron), and (5) Linux-first design. Heynote comes closest on features 1-2 but is Electron-based and not workspace-aware. Terminal tools (Thoth, Snip) lack GUI. GUI tools lack CLI integration.

## 6. Developer Sentiment: Electron vs Tauri

### Electron Backlash (2025-2026)

The backlash is real and intensifying:

- "Chat clients use hundreds of megabytes of RAM just to sit idly in the background, note-taking applications use the same amount of RAM as Photoshop did in the past"
- Discord (Electron): normal usage under 1 GB but **reaching 4 GB** under real-world conditions
- WhatsApp moved from lightweight UWP (100 MB idle) to a Chromium wrapper
- Microsoft Teams moved from Electron to WebView2 but **still idles at ~1 GB RAM**
- A filed GitHub issue reports: "Electron-based apps cause a huge system-wide lag on macOS 26"
- RAM prices rising while Electron apps use more RAM creates growing user frustration

**Root cause**: "For every one native developer, there are eight web developers" -- companies choose Electron because talent is available, not because users prefer it.

### Tauri's Rise

**Adoption**: Up **35% year-over-year** after Tauri 2.0 (late 2024). Community: **17,700+ Discord members**.

**Performance benchmarks (2026)**:
- Memory: Electron apps idle at **200-300 MB**; Tauri sits at **30-40 MB** (50% less memory)
- App size: Electron apps typically **100+ MB**; Tauri apps usually **under 10 MB**
- Startup: Electron **1-2 seconds**; Tauri **under 0.5 seconds**
- 2026 benchmark: Tauri showed **40% improvement in startup time** and **30% reduction in memory usage**

**Tauri 2.0 capabilities**: Now targets mobile alongside desktop (Windows, macOS, Linux, Android, iOS from single codebase). Distribution to Apple App Store, Google Play, Microsoft Store, Flathub, Snapcraft, AUR, and more.

**Developer sentiment**: Tauri viewed as "the future of lightweight desktop apps" while Electron remains "the king of ecosystem and maturity." The main Tauri barrier is Rust learning curve for teams unfamiliar with it.

## 7. Local-First / Privacy-First Software Movement

### Movement Maturity

The local-first movement has gone from academic concept to mainstream developer concern:

- A "Local-First Manifesto" signed by **500 influential software architects** signaled a decisive move away from cloud-only paradigms
- **FOSDEM 2026** hosted its **first-ever Local-First devroom** (Feb 1, 2026) with **22 talks by 25 speakers** covering CRDTs, sync engines, E2EE, and UX patterns for offline-first apps
- **Open Local First** (openlocalfirst.org) has emerged as a community organizing body

### Driving Forces

- Record-breaking data breaches in 2025 exposed billions of records from centralized data stores
- Developer discomfort with "the rentier model of SaaS" where functionality that once lived on machines is behind API calls and subscriptions
- Growing pushback against apps requiring always-on connectivity

### Key Projects & Tools in the Ecosystem (2025-2026)

**CRDT Libraries (mature)**:
- **Yjs** -- Dominant JS CRDT implementation, real-time sync, automatic merge
- **Automerge 2** -- Production-ready, now with Keyhive integration for E2EE
- **Loro** -- High-performance Rust CRDT library (memory/CPU optimized, not yet production-ready)

**Sync Engines**:
- **PowerSync** -- Most production-ready for existing Postgres/MongoDB/MySQL databases. Syncs to local SQLite on client.
- **ElectricSQL** -- Postgres-focused, active-active sync to client-side SQLite. Featured at FOSDEM 2026 (TanStack DB integration).
- **Jazz** -- CRDT framework with E2EE and permissions (featured talk at FOSDEM 2026)

**Local-First Applications**:
- **Plane** -- Collaborative wiki built with Yjs, presented at FOSDEM 2026 as a production case study
- **Radicle** -- Local-first code collaboration (Git-compatible, decentralized)
- **NextGraph** -- E2EE sync engine with graph DB and reactive ORM
- **SQLRooms** -- Local-first analytics with DuckDB, collaborative canvas, and Loro CRDT sync
- **Fireproof** -- Pure JS database, zero dependencies, CRDT support, E2EE

**Infrastructure**:
- **Willow** -- Protocols for peer-to-peer data sync
- **js-libp2p / IPFS / OrbitDB** -- P2P networking stack for local-first apps
- **Seed Hypermedia** -- "The Future of Digital Sovereignty"

### Developer Sentiment

Developers can leverage end-to-end encryption more easily in a local-first environment, with sync layers only seeing encrypted blobs. The ecosystem has "finally caught up to the philosophy" in 2026 with mature tooling from PowerSync, ElectricSQL, and Automerge.

## Strategic Implications for Notey

Based on this research, several market signals are strongly favorable for Notey's positioning:

1. **Tool fatigue is acute**: Developers juggle 14 tools and lose 6-15 hours/week to context switching. A tool that captures without switching context (global hotkey, CLI integration) directly addresses the #1 productivity complaint.

2. **Electron backlash creates an opening**: Heynote (the closest competitor) is Electron-based. Building on Tauri gives Notey a 5-10x advantage on memory (30-40 MB vs 200-300 MB) and app size (under 10 MB vs 100+ MB). Tauri adoption is up 35% YoY -- developers actively want this.

3. **Local-first is no longer niche**: With FOSDEM dedicating a devroom, 500-architect manifestos, and mature CRDT tooling, "local-first" has become a positive positioning signal, not a limitation to apologize for.

4. **The scratchpad category is validated but unfilled**: Heynote proved massive demand (978 HN points, 5.2k GitHub stars) but is broadening toward "power users" with images/diagrams. No tool combines global hotkey + CLI + workspace awareness + native performance. That gap is Notey's.

5. **Terminal/CLI integration is underserved**: New tools like Thoth and Snip show demand for terminal-native note capture, but they lack GUI counterparts. Notey's dual CLI+GUI approach is differentiated.

6. **Linux-first is underserved**: LinNote (KDE/Qt6) shows Linux developers want native scratchpads. Most tools are Mac-first. Notey's Linux-first approach targets an underserved segment.
