## Intent

**Problem:** Users have no way to delete a note. Deletion must never destroy data: a deleted note has to move to a recoverable trash, vanish from active lists, surface a non-blocking confirmation toast, and not leave a stale editor tab open. The backend soft-delete already exists but is unreachable — there is no command, no UI trigger, and no toast system.

**Approach:** Expose the existing `trash_note` service as a Tauri command, add a minimal reusable toast system, and wire a command-palette "Move to Trash" action that soft-deletes the active note, closes its tab (activating an adjacent tab or the empty state), refreshes the note list, and shows the toast "Note moved to trash".

## Boundaries & Constraints

**Always:**
- Soft-delete only — set `is_trashed = 1` and `deleted_at` to an ISO8601 timestamp. Never hard-delete; never remove rows from `notes`.
- All IPC via tauri-specta generated bindings (`commands.trashNote`) — never raw `invoke()`. Regenerate `src/generated/bindings.ts` by running backend tests after registering the command.
- The command handler stays thin and synchronous, delegating to the existing `services::notes::trash_note`.
- The toast is non-blocking: it must not steal focus, capture pointer events over the editor, or interrupt typing. Auto-dismiss after 3s, bottom-right.
- Closing the trashed note's tab reuses the existing `closeTab` adjacency logic (right neighbor → left → empty state). Do not reimplement tab selection.
- Follow project conventions: named Zustand actions only, per-feature store, `camelCase` IPC fields, JSDoc on exported symbols, no barrel files.

**Ask First:**
- Adding any third-party dependency (e.g. `sonner`) for toasts — the app is offline/minimal; this spec builds a small in-repo toast instead.
- Adding a global keyboard shortcut for delete (collides with editor keys / Epic 7 hotkey config) — out of scope here.

**Never:**
- No confirmation dialog (soft-delete is reversible — confirmation is Story 5.3's permanent-delete concern).
- No Trash view, restore, permanent delete, or auto-purge (Stories 5.2–5.4).
- No manual mutation of `notes_fts` — the existing triggers handle it.
- No changes to the already-correct `trash_note` service SQL or the `list_notes` trashed-filter.

## I/O & Edge-Case Matrix

| Scenario | Input / State | Expected Output / Behavior | Error Handling |
| -------- | ------------- | -------------------------- | -------------- |
| Trash active note (happy) | Active tab shows note N (not trashed) | `trash_note(N)` sets `is_trashed=1`, `deleted_at=now`; N's tab closes; adjacent tab activates (or empty state); note list refreshes without N; toast "Note moved to trash" (3s) | N/A |
| Trashed note not open in a tab | Note N exists, no tab for N | N soft-deleted; list refreshes without N; toast shown; tab state unchanged | N/A |
| No active tab | No tabs open | No command invoked, no toast, no state change | No-op (silent) |
| Note missing / already trashed | `trash_note(N)` where N absent or already `is_trashed=1` | Backend returns `NotFound`; no tab closed; list unchanged | Frontend logs error, shows error toast "Couldn't move note to trash", returns null |
| Last tab trashed | Single tab shows N | Tab closes; `activeTabIndex` → null; editor shows empty state | N/A |

**Acceptance Criteria:**

- Given a registered `trash_note` command, when the frontend calls `commands.trashNote(id)` on a non-trashed note, then `is_trashed` becomes 1 and `deleted_at` is a current ISO8601 timestamp, and the note no longer appears in `listNotes` results.
- Given the active note is soft-deleted, when the operation succeeds, then a non-blocking bottom-right toast "Note moved to trash" appears and auto-dismisses after 3s while the user can keep typing in another note.
- Given the trashed note was open in a tab, when the trash completes, then that tab is closed and an adjacent tab becomes active, or the empty state shows if it was the only tab.
- Given the palette "Move to Trash" command is invoked with no active tab, when it runs, then nothing happens (no command call, no toast, no state change).
- Given `trash_note` is invoked for a missing or already-trashed note, when it runs, then the backend returns `NotFound`, the note list and tabs are unchanged, and an error toast is shown.
- Given the new command, when `cargo test` runs, then `acl_tests` passes with `allow-trash-note` and `bindings.ts` contains `trashNote`.

## Design Notes

The backend service and its filter logic already exist and are tested — this story is mostly *exposure + UI*. Resist re-implementing soft-delete SQL.

Toast id generation uses a module-level incrementing counter (deterministic, test-friendly) — not `Date.now()`/`Math.random()`. Auto-dismiss lives in the store's `addToast` so a toast self-expires regardless of which trigger created it; tests drive it with `vi.useFakeTimers()`.

Cross-store calls via `useStore.getState()` are an established pattern here (see `NoteListPanel` calling `useTabStore.getState()`), so `trashNote` calling the tab store and `trashActiveNote` calling the workspace + toast stores is consistent. The editor swap still lives in `EditorPane.tsx`, but same-index active-note replacements (for example, closing the active tab when the right neighbor shifts into its slot) must track note identity as well as `activeTabIndex` so the stale note is not left onscreen.

Toast skeleton:

```ts
let nextId = 0;
addToast: (message, durationMs = 3000) => {
  const id = nextId++;
  set((s) => ({ toasts: [...s.toasts, { id, message }] }));
  setTimeout(() => get().dismissToast(id), durationMs);
},
```
