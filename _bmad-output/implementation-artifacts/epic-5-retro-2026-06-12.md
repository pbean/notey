## Successes

1. **The +91 test reversal.** Epic 4's headline worry was thin frontend test growth (+7 across 9 stories). Epic 5 added **91** (trash store 17, toast 10, TrashPanel 14, ConfirmDeleteDialog 10, export actions 10, plus more). This effectively confirms Epic 4's jsdom-limit hypothesis: lifecycle/export logic (store actions, toasts, a dialog) is far more jsdom-testable than drag-and-drop and multi-instance CodeMirror were. The concern self-corrected because the *work* changed shape.

2. **The frozen-spec contract did real work.** Every story carried a `<frozen-after-approval>` intent block. The review ledgers are full of dismissals that cite it — *"the frozen spec explicitly requires regenerated bindings," "the spec explicitly defines `0` as immediate purge," "the frozen design chose collect-then-write."* The contract stopped the adversarial reviewer from re-litigating settled decisions and held scope.

3. **Adversarial review caught real security/correctness bugs every story — caught, not shipped.** The filesystem epic got a genuinely strong path-containment review: file-overwrite clobbering (5.5), Windows device-name / `..` / byte-length sanitization (5.5), control-char escaping in YAML values (5.5), **symlink-target escape** of the chosen directory (5.6), and a `BufWriter` drop swallowing a final write failure → truncated-but-"successful" JSON (5.6).

4. **The timestamp-comparability trap was caught in *design*, not as a bug (5.4).** Storing `deleted_at` as RFC3339 while computing the purge cutoff with SQLite's `datetime('now','-30 days')` would silently mis-purge near the boundary (different string formats sort differently). The spec computed the cutoff in Rust with the same `to_rfc3339()` formatter and used strict `<` so a note exactly 30 days old survives ("at least 30 days"). Prep-de-risks applied to a data trap.

5. **Epic 4's debt was retired *before* Epic 5 started.** Both committed action items shipped — startup config application (DW-80) and the toggle concurrency guard (DW-81) — plus three bonus hardening items the cleanup surfaced: boot-window toggle race (DW-82), `prefers-color-scheme` resolution for `theme: 'system'` (DW-83), and `floating`→`comfortable` default alignment (DW-84).

6. **Reuse held the line.** New commands mirrored the `trash_note` shape exactly (lock via `recover_poisoned_db` → testable service → rowcount / `NotFound`). `ConfirmDeleteDialog` reused the existing Base UI `Dialog` (focus trap, scroll lock, Esc-to-close, backdrop free) — only overriding `role` to `alertdialog`. Most of the epic was *exposure + UI* over backend that already existed.

## Challenges

1. **The concurrency guard was hand-rolled six times — and the same bug was caught twice.** Every story needed an in-flight guard: `isTrashingNote` (5.1), per-id `restoringIds` Set (5.2), `deletingIds` Set (5.3), `isExporting` (5.5, 5.6) — on top of `isCreatingNote` from Epic 4. The **exact same defect** — *"deduped null result turns into a false 'Couldn't…' failure toast"* — was caught independently in **both 5.2 and 5.3** review, plus 5.3's *"in-flight completion closes a newer dialog."* A cross-cutting concern touching 6 stories was patched story-by-story instead of abstracted once.

2. **Async staleness / identity bugs, again.** 5.1 *"re-run the editor swap when the active note changes at the same tab index"*; 5.2 *"older trash loads can win races," "removing the selected last row leaves `selectedIndex` stale"*; 5.3 *"in-flight delete completion closes the wrong dialog."* These are the same **boundary/identity/staleness** class Epic 4 flagged — not logic bugs. Targeted property/boundary tests catch them earlier than review.

3. **E2E: the fifth epic with no new coverage.** The only E2E is the Epic 1 capture-loop + window-management suite (`e2e/run.mjs`, 7 tests via tauri-driver). Search, multi-tab, command palette, trash lifecycle, and now **export that writes real files to the real filesystem** have zero E2E — the native pickers, the file write, and the path-containment we spent heavy review effort hardening are verified by hand only. bmad-auto now drives whole epics unattended, which (per Epic 4's own insight) raises — not lowers — the stakes on end-to-end coverage. This was explicitly deferred to "revisit at Epic 5 retro." **Resolved this retro** → folded into Action Item 1 (full test-design + backfill before Epic 6).

4. **Retro-item tracking drift.** Epic 4 action items 1 & 2 are *done in code* (DW-80/81, committed 2026-04-13) but their lines in `sprint-status.yaml` still read `backlog`. The work shipped; the tracker wasn't flipped on close-out.

5. **The permission-TOML auto-generation gotcha recurred on every new-command story.** Each of `trash_note`, `restore_note`, `list_trashed_notes`, `delete_note_permanently`, `export_markdown`, `export_json` needed the manual TOML check (known Tauri v2 issue, documented in project-context). Epic 6 adds more commands — the manual dance continues.

## Key Insights

1. **Cross-cutting concurrency-with-feedback deserved an up-front abstraction.** Exactly the Epic 4 lesson (overlay coordination) in a new guise. When a concern touches 3+ stories, design the shared abstraction before the first consumer, not after the sixth. Current (June 2026) guidance confirms the ref/in-flight-flag-reset-in-`finally` primitive is correct; the recurring *bug* is the **return semantics** — a deduped call returning ambiguous `null` that callers misread as failure. The singleflight / `createSharedPromise` pattern (concurrent callers await the *same* in-flight result) structurally removes the false-null. → Action Item 2.

2. **bmad-auto continues to raise the stakes on shifted-left quality.** A full unattended epic at the same quality bar is a real milestone — but it means the E2E gap and any thin automated coverage are now higher-priority than when humans reviewed every diff. The test-backfill pass (Action Item 1) is the direct response.

3. **Prep-de-risks-complexity, confirmed again — now for data traps, not just hard components.** The RFC3339-vs-SQLite-datetime analysis in 5.4's spec is the same move as the CodeMirror/FTS5 research docs: surface the trap in design so it never becomes a runtime bug.

4. **The frozen-intent spec is a force multiplier for adversarial review.** It lets the reviewer be maximally cynical on *implementation* while a clear contract prevents wasted cycles re-arguing *intent*. Worth keeping as standing practice.

5. **The riskiest epic so far is next, and it lands on our thinnest integration-test muscle.** Epic 6 introduces a second process (CLI) over an IPC socket with a scored injection risk (RISK-006). Building integration-test capability *before* it (Action Item 1) is deliberate sequencing, not gold-plating.
