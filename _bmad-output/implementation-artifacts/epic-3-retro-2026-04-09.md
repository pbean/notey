## Challenges

1. **Frontend test isolation gaps (recurring, 3/5 stories)** — Store state bleeding between tests, `.cm-content` DOM nodes leaking, workspace store not reset. Each frontend story (3.3, 3.4, 3.5) had test isolation issues caught in review. Root cause: no shared test setup/teardown pattern — each test file reinvents its own cleanup.

2. **Async error handling omissions (recurring, 3/5 stories)** — Story 3.3: no try/catch around `commands.searchNotes()`. Story 3.4: unhandled promise rejection in `openNote()`, stale closure in keydown handler (not memoized with `useCallback`). Story 3.5: whitespace trim inconsistency between `handleInput` and `handleScopeToggle`. All caught by review, not by development.

3. **Story 3.1 scope explosion via review** — Scoped as "migration + tests only," but 3 review passes drove: workspace validation hardening (trim, path validation), error propagation fixes (workspaceError, notesError state fields), `rebuild_fts_index` command, format guard in `loadNote`, upsert_workspace trim fix, and notesError UI consumer. All valuable, but the story record is hard to interpret because half the review findings aren't about FTS5.

4. **Permission TOML auto-generation still failing** — Recurring from Epic 2. Story 3.2 required manual TOML creation. Documented in project-context.md, but the underlying Tauri v2 issue persists.

5. **Test quality concerns** — Story 3.4's focus trap tests were "vacuous" (jsdom doesn't move focus on `fireEvent.keyDown`, so assertions passed trivially). Story 3.3's `parseSnippet` regex was greedy. These are the kind of issues that tooling should catch before review.

---

## Key Insights

1. **Code review is our strongest quality gate — but we're over-relying on it.** Review consistently catches preventable issues (test isolation, error handling). Shifting these left with tooling (shared test helpers, ESLint rules) will make review more effective by letting it focus on architectural and design concerns rather than recurring mechanical issues.

2. **Retro action item tracking works — 100% follow-through proves the system.** The sprint-status tracking added after Epic 2 is the meta-process improvement that makes all other improvements visible and accountable. Keep it.

3. **Research documents de-risk complex stories.** The FTS5 research document from Epic 2's retro enabled Story 3.1 to start with exact migration SQL on day one. Apply the same pattern to CodeMirror multi-instance for Epic 4.

4. **The "fix everything" approach is correct for greenfield.** Pinkyd confirmed: review-driven scope expansion is the right call. Review finding tags ([Patch], [Defer], [Decision]) provide traceability even when diffs are broader than story titles.

5. **Agent model may matter.** Story 3.1 (Sonnet) required 3 review passes; Stories 3.2-3.5 (Opus 4.6) averaged ~1 pass. Correlation, not proven causation, but worth noting for story creation decisions.

## Technical Debt

### Outstanding (Carried Forward)

1. **`scopeFilter` doesn't resync on workspace switch** — Search scope persists across workspace changes without resync (deferred from Story 3.5 review). Design decision per AC 6, but could confuse users.
2. **Double-canonicalize in workspace service** — `resolve_workspace` -> `create_workspace` chain redundantly canonicalizes paths (pre-existing, internal function).
3. **FTS5 no down migration** — Pre-existing pattern, no migrations have down. Not blocking but increases migration complexity long-term.
4. **Specta i64-to-BigInt precision limitation** — JS `number` loses precision beyond 2^53. Specta 2.0.0-rc.24 has no BigInt mapping. Only materializes above 9 quadrillion IDs. Not practical to fix now.
5. **Whitespace trim inconsistency** — `handleInput` checks `=== ''` while `handleScopeToggle` checks `.trim()` in SearchOverlay (deferred from Story 3.5 review).
6. **E2E test coverage still at zero** — 3 epics, zero automated E2E tests. Deferred from Epic 1 and 2 retros.
