## Successes

1. **100% story completion** — all 14 stories delivered, complete capture loop operational
2. **tauri-specta compile-time IPC safety** — zero frontend-backend type mismatches across all 14 stories. Investment validated.
3. **CodeMirror Compartment pattern** — discovered in Story 1.8, enabled clean format toggle in Story 1.10 without recreating editor view or losing undo history
4. **Disciplined deferred work cleanup** — 6 tech debt items identified during Stories 1.11-1.14, dedicated hardening pass resolved 5 of 6:
   - Store reset (`resetNote()` action)
   - Window focus on hidden surface (`useWindowFocus` hook)
   - Server-side timestamps (replaced client `Date.now()` with server `updatedAt`)
   - Live hotkey re-registration with rollback
   - Atomic config write (temp+rename pattern)
   - Mutex scoping (lock released before filesystem I/O)
5. **Strong backend integration tests** — 10+ tests covering PRAGMAs, WAL mode, CRUD, trash filtering, temp-file DB isolation per test
6. **`flushSave()` extraction** — module-level function enabled clean Esc-dismiss save flushing

---

## Challenges

1. **Specta nightly Rust requirement** — `tauri-specta v2.0.0-rc.24` requires nightly Rust, conflicting with project-context mandate for stable. Resolved by pinning `nightly-2026-04-03` in `rust-toolchain.toml`. Not a planning failure — discovery during implementation is inevitable with RC-stage dependencies.

2. **Error variant conflict** — Spec listed 5 `NoteyError` variants, project-context listed a different 5. Reconciled to 6 variants (`Database`, `NotFound`, `Workspace`, `Io`, `Validation`, `Config`) for forward-compatibility.

3. **Window lifecycle complexity** — Stories 1.11-1.14 generated 6 deferred items. Pre-create hidden, never destroy, hide on Esc, prevent close on X, focus-on-hidden-surface handling — significant complexity in a single cluster.

4. **Test design workflow bypassed** — The biggest systemic gap. A thorough test design existed (`_bmad-output/test-artifacts/test-design/notey-handoff.md`) with clear workflow: TEA Test Design -> ATDD generates acceptance tests -> Implementation -> Test Automation -> Trace validation. Steps 3, 5, and 6 were skipped entirely. No failing acceptance tests were generated before implementation. No trace matrix was produced.

---

## Key Insights

1. **A quality plan without enforcement mechanisms gets bypassed under delivery pressure.** The test design handoff document mapped specific test IDs to story scopes, defined infrastructure prerequisites, and specified phase transition gates. None of this was consumed during story execution.

2. **Discovery during implementation is inevitable with RC-stage dependencies.** The team's resolve-and-document response (nightly pin, error variant reconciliation) was effective. This is not a planning failure to fix — it's a reality to acknowledge.

3. **Disciplined debt cleanup is a pattern worth repeating.** Identifying deferred items during implementation and resolving them in a dedicated pass worked well. Apply this pattern to future epics.

4. **Test infrastructure must be treated as prerequisite work, not afterthought.** Test data factories, CI pipeline, and E2E framework setup should precede story implementation.
