## Challenges

1. **Retro action item tracking gap** — Epic 1 produced 8 action items. Most were completed by agents, but there was zero visibility. Pinkyd gave clear direction, agents executed, but nobody reported back or tracked completion. The sprint-status file tracks stories but not retro commitments. This is the **biggest systemic gap** identified.

2. **Console.error logging gaps** — code review flagged missing error logging in async store actions in 3 of 6 stories (Stories 2.4, 2.5, 2.6). Systematic habit gap in frontend error handling.

3. **Permission TOML auto-generation failures** — Stories 2.1 and 2.2 both discovered that Tauri v2 doesn't always auto-generate permission files. The workaround (manual creation) was rediscovered twice before being documented in story intelligence.

4. **base-ui vs Radix API mismatch** — Story 2.4 discovered that `@base-ui/react/menu` uses `onClick` instead of Radix UI's `onSelect` convention, causing test failures in jsdom.

5. **Borrow checker edge case** — Story 2.5 hit a `stmt.query_map()` borrowing issue in match arms. Resolved by binding result to local variable before block end.

---

## Key Insights

1. **Process improvements embedded in story workflow get done; standalone infrastructure tasks don't.** The 3 retro commitments we completed (test design in stories, compliance checks, project-context update) were woven into the story creation/review workflow. The items that needed separate execution (CI pipeline, test factories) appeared to be deferred — though investigation revealed agents had actually completed them without reporting back.

2. **Retro action items need the same tracking visibility as story status.** Without a forcing function, commitments made in retrospectives have no accountability mechanism. Adding retro items to sprint-status.yaml fixes this.

3. **Knowledge compounds well within an epic (story intelligence sections) but not across epics (retro items lost).** The Previous Story Intelligence sections were a major win for intra-epic learning. Cross-epic learning requires a different mechanism — tracked action items and documented research.

4. **Code review is consistently valuable.** Zero CRITICAL issues across 6 stories means the implementation quality is high, but MEDIUM findings in 5/6 stories means review catches real improvements. The recurring console.error pattern suggests a checklist item or linting rule would help.
